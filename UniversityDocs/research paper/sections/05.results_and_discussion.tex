\section{Results and Discussion}
\label{sec:results_and_discussion}

This section presents experimental results for accuracy, model sizes, latency, and memory usage, followed by interpretation and discussion of practical trade-offs when deploying Mamba on MCUs.

\subsection{Model accuracy}

We first trained models on a development PC using a custom PyTorch implementation of the Mamba block. Training produced validation accuracies of 98.6\% for HAR and 90.09\% for the KWS task.

We then converted the models to TFLite and evaluated accuracy before and after quantization. Results are shown in \autoref{tab:accuracy-results} for both the original (full) model and the split model.

It is often assumed that storing model data in flash is much slower than storing it in RAM. We measured a simple array-sum benchmark with data in DRAM versus flash on the ESP32 and found flash access ~2.15x slower for this task (far smaller than typical desktop disparities). Note that TFLM copies relevant arrays from ROM to RAM on tensor initialization, so this initial-copy overhead is separate from steady-state inference behavior.

\subsection{Embedded Performance}

The next important metrics are the inference time (latency) and RAM usage on the microcontroller.

We started by assessing the impact each operation had on the latency (\autoref{fig:operation_performance}). This motivated us to limit usage of exponentiation in the inference, as it was the easiest to eliminate, and this optimization has already been implemented by similar works \cite{kimEMambaEfficientAcceleration2025}. This led us to create 3 similar Mamba models: \textit{float noopt}, \textit{float}, \textit{float opt}. The noopt model does not replace any of the exp operations, the float model replaces the SiLU and Softmax operations with ReLU, and the float opt in addition to the function replacement replaced \texttt{A\_bar = torch.exp(delta\_t * A\_reshaped)} line with Euler Discretization \texttt{A\_bar = 1.0 + delta\_t * A\_reshaped}. The latency of those models can be seen on \autoref{fig:float_opt}. We also measured accuracy of those models, which can be seen in \autoref{tab:accuracy_float}. Because of the similar accuracy for the \textit{float noopt} and \textit{float} models, as well as the noticeable accuracy decrease for the quantized \textit{float opt} model, we decided to use \textit{float} as the baseline for future optimizations\footnote{We later found out that depending on the seed the accuracy decrease for the \textit{float opt} might not occur, and achieved 94.81\%, 94.71\% accuracy results on one testing run}.

\subsection{Discussion}
\label{sec:discussion}

We show that quantization of small mamba models is viable. The drop in accuracy depends on the dataset, and as pointed out in the MambaQuant paper mamba quantization can be difficult. This can be seen for the KWS models, which lose nearly 4 percentage points of accuracy when quantized. However, even this loss will be acceptable for many applications.

The split of the model allows us to run more complex models, as we did not manage to run the KWS model without this operation. This was partially caused by the limitations of the TFLM, with the concatenate operation allowing only up to 10 vectors to be connected at once, while for KWS it would need to concatenate 51 time steps. A surprising outcome of the split was that it also reduces the peak memory usage of the model. Our hypothesis is that this is caused by the allocation style used by TFLM, which is based on a greedy algorithm, and we suspect the split was able to improve this decision. If true, splitting might not always have the same effect.

We also find that the split model is more sensitive to quantization, possibly for reasons such as:

\begin{itemize}
  \item static quantization achieving a better fit for the fully unrolled graph
  \item numerical error accumulating more quickly in our SSM loop implementation
\end{itemize}

The unrolled graph can use different scales and zero points for different instances of the same operation, which may explain why the full (unrolled) model performs better under static quantization.

Our Mamba models, though smaller than state-of-the-art large models—achieve satisfactory accuracy: nearly 94\% on HAR and ~89\% on KWS in the configurations tested, demonstrating Mamba's potential for compact tasks.

We observe that much of the latency arises from SUM, MUL, ADD, and EXP operations in the SSM layer. From a latency perspective, the SSM does not map efficiently to TFLM as-is, in part because of costly exponentiation on MCUs. Implementing a custom TFLM SSM kernel (possibly quantized) would likely yield substantial speedups; time constraints prevented us from implementing this in the current work.

We also see that the split while it allows for running larger inference models has its drawbacks in form of latency.

\subsection{Limitations}

There are several limitations. Tooling for deploying Mamba on MCUs is immature, and time constraints limited our exploration.

We evaluated only two popular TinyML datasets, which do not cover use cases such as image/video tasks where Mamba may also be useful. We tested FLOAT32 and INT8 formats; exploring lower bit widths could be valuable but is currently limited by TFLM operator support.

Results were gathered on a single MCU family (ESP32); different vendors' hardware acceleration choices may yield substantially different latency and performance characteristics. In our experiments we encountered bugs with the ESP32's implementation of some optimizations, which also makes our research less reliable.

Finally, our split approach complicates deployment because it requires model-aware inference code. It also drastically increases the latency of the model. Implementing the SSM as a custom, quantized TFLM kernel would preserve deployment simplicity while enabling the same memory/time trade-offs; we leave that for future work.
